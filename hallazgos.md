# Brechas de Seguridad y Fallos Críticos del Sistema - Entrafesa

A continuación se detalla la matriz de hallazgos específicos identificados en la revisión del código fuente de los módulos de NestJS (`api`, `api-sales`) e Inteligencia Artificial (`ia`).

| Módulo / Archivo                                     | Tipo de Riesgo                                   | Detalle Específico del Fallo / Brecha                                                                                                                                                                                                                                                     | Propuesta de Solución                                                                                                                                                        | Impacto            |
| :--------------------------------------------------- | :----------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------------- |
| **`notifications.controller.ts`**                    | 🚨 **Escalación de Privilegios (BFLA)**          | Los endpoints `/all-alerts` y `/massive` heredan el decorador de clase `@Auth([RoleEnum.USER, RoleEnum.ADMIN, RoleEnum.SELLER])`. Esto permite que un cliente regular (`RoleEnum.USER`) consulte las notificaciones de todos los usuarios o envíe un spam masivo a toda la base de datos. | Restringir el acceso a estos endpoints específicos mediante un decorador a nivel de método que exija únicamente `RoleEnum.ADMIN`.                                            | **Alto** (Crítico) |
| **`notifications.service.ts`** (Línea 153)           | 💥 **Denegación de Servicio (OOM)**              | El método `sendMassiveAlert` realiza un `profileRepository.find()`, cargando absolutamente todos los perfiles de la base de datos a memoria de golpe antes de mapearlos. En producción, esto provocará el desbordamiento de la memoria RAM (OOM) y la caída del servicio.                 | Implementar paginación (offset/limit) o procesamiento por lotes (chunks/streams), idealmente delegando esta tarea a un queue worker en segundo plano (ej. BullMQ con Redis). | **Alto**           |
| **`webhook.py`** (Línea 47)                          | 🔒 **Falta de Autenticación de Webhook**         | El endpoint público `/webhook` de FastAPI no valida firmas, tokens ni procedencia de Chatwoot. Cualquier atacante externo puede enviar peticiones POST simulando mensajes de chat.                                                                                                        | Implementar la verificación del token secreto en los headers (`X-Chatwoot-Signature` o similar) provisto por Chatwoot para autenticar el webhook.                            | **Medio**          |
| **`openai_client.py`** (Línea 17)                    | 💾 **Pérdida de Estado y Fuga de Memoria**       | Las sesiones del chat se guardan en un diccionario en memoria `CHAT_SESSIONS = {}`. Al escalar el servicio horizontalmente (múltiples réplicas/contenedores), el historial de chat será inconsistente y además causará fugas de memoria con el tiempo.                                    | Migrar el almacenamiento de sesiones y el historial de chat a una base de datos de clave-valor rápida como Redis con un tiempo de expiración (TTL).                          | **Medio**          |
| **`chatwoot_service.py`** (Línea 36)                 | ⏳ **Bloqueo de Hilos (Falta de Timeout)**       | La petición HTTP `requests.post(...)` no define un parámetro de `timeout`. Si Chatwoot sufre lentitud o caída, el hilo de ejecución se bloqueará indefinidamente, agotando el pool de conexiones de FastAPI.                                                                              | Especificar siempre un tiempo de espera explícito en la llamada: `requests.post(url, json=payload, headers=headers, timeout=5.0)`.                                           | **Medio**          |
| **`sales.service.ts`** (Línea 90)                    | 🎲 **Colisión de IDs por Aleatoriedad Insegura** | La generación del código de referencia del hotel utiliza `Math.random().toString(36).substring(2, 15)`. `Math.random()` no es criptográficamente seguro y generará duplicados bajo cargas transaccionales concurrentes.                                                                   | Utilizar identificadores únicos universales (UUID v4) mediante la librería estándar `crypto` de Node.js o una secuencia autoincremental de base de datos.                    | **Medio**          |
| **`auth.service.ts`** y **`public-auth.service.ts`** | 👤 **Enumeración de Usuarios**                   | Los métodos de login lanzan errores diferenciados (`NotFoundException('User not found')` e `Invalid password`). Esto permite a un atacante automatizar consultas para mapear qué números de DNI o documentos están registrados.                                                           | Retornar un mensaje de error genérico para ambos escenarios, como `'Credenciales incorrectas'` o `'Usuario o contraseña inválidos'`.                                         | **Bajo**           |
| **`main.ts`** (NestJS Core, Línea 11)                | 🌐 **Configuración de CORS Permisiva**           | Se configura `app.enableCors('*')` permitiendo peticiones desde cualquier origen sin restricciones.                                                                                                                                                                                       | Restringir el acceso CORS únicamente al dominio configurado de la aplicación (`APP_URL`) en entornos de producción.                                                          | **Bajo**           |
